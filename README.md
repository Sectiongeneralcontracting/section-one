# Section One — Construction Management Platform

نظام إدارة متكامل لشركة Section General Contracting، مبني بمعمارية **Modular Monolith** قابلة للتوسع. حاليًا فيه 7 موديولات شغالة ومتكاملة مع بعض:

1. **الأرباح والشركاء** — مشاريع، عملاء، مصروفات، شركاء ونسب أرباح
2. **العقود والمستخلصات** — عقود، جداول كميات (BOQ)، مستخلصات بحساب تلقائي
3. **الموردين والمشتريات** — موردين، أوامر شراء بـ workflow
4. **المخازن** — مخازن وأصناف وحركات وارد/منصرف بأرصدة محسوبة تلقائيًا
5. **المعدات** — تخصيص للمشاريع وسجل صيانة
6. **الموارد البشرية والرواتب** — موظفين ورواتب بحساب تلقائي
7. **الحضور والانصراف** — تسجيل يومي مربوط بحساب الرواتب

بالإضافة لموديولات النظام العامة: المستخدمون والصلاحيات، التقارير (بما فيها تقارير تنفيذية شاملة)، الإعدادات، Backup/Restore، والإشعارات.

كل موديول جديد = Prisma models + مجلد `src/app/<module>/` + `src/app/api/<module>/` منفصل، من غير ما يأثر على اللي موجود.

## Stack
- Next.js 14 (App Router) + TypeScript
- PostgreSQL (Vercel Postgres) + Prisma ORM
- NextAuth (Credentials) — أدوار: Admin / Manager / Viewer
- Tailwind CSS — Material Design 3 style, دعم كامل RTL/LTR

## ما تم بناؤه — Module 2: العقود والمستخلصات
- قاعدة بيانات: `Contract` (مرتبط 1:1 بالمشروع — رقم عقد، تاريخ توقيع، مدة تنفيذ، نسبة ضمان حسن التنفيذ، نسبة وقيمة الدفعة المقدمة)، `BoqItem` (جدول الكميات: كود، وصف، وحدة، كمية، سعر وحدة)، `PaymentCertificate` (مستخلص بنسبة إنجاز تراكمية، مع Workflow: مسودة → مقدَّم → معتمد → مصروف)
- **حساب المستخلص تلقائي بالكامل**: قيمة الفترة = (نسبة الإنجاز التراكمية × قيمة العقد) − آخر مستخلص معتمد، وبيخصم منها ضمان حسن التنفيذ + استرداد الدفعة المقدمة (بيتوقف تلقائيًا عند استرداد كامل القيمة) + الضريبة، ليطلع صافي المستحق. المنطق ده في `src/lib/certificates.ts` ومغطى باختبارات Vitest كاملة (حالات: أول مستخلص، مستخلص تراكمي، استرداد جزئي/كامل للدفعة المقدمة، ضريبة)
- Workflow صلاحيات: التقديم بصلاحية Manager، الاعتماد والصرف بصلاحية Admin فقط، ومنع أي قفزة في الحالة (مينفعش تعتمد مستخلص لسه مسودة)
- API: `/api/contracts` (+`/[id]`) `/api/contracts/[id]/boq` `/api/boq-items/[id]` `/api/contracts/[id]/certificates` `/api/certificates/[id]` (تقديم/اعتماد/صرف)
- شاشات: **العقود** (قائمة + إضافة عقد لمشروع بدون عقد)، **تفاصيل العقد** (جدول كميات BOQ قابل للإضافة والحذف + جدول مستخلصات بيحسب نفسه تلقائيًا مع أزرار الـ Workflow)

## ما تم بناؤه — Module 3: الموردين والمشتريات
- قاعدة بيانات: `Supplier` (اسم، تصنيف: مواد/معدات/مقاولي باطن/خدمات، بيانات تواصل، رقم ضريبي)، `PurchaseOrder` (مرتبط بمورد ومشروع اختياريًا، Workflow: مسودة → معتمد → مُستلم → مصروف، أو إلغاء)، `PurchaseOrderItem` (بنود الأمر: وصف، وحدة، كمية، سعر وحدة)
- فورم إضافة أمر شراء ببنود متعددة ديناميكية (إضافة/حذف بند) مع حساب الإجمالي لحظيًا
- Workflow بصلاحيات: التقديم/الاعتماد بصلاحية Manager، تسجيل الصرف بصلاحية Admin فقط، ومنع أي انتقال حالة غير منطقي (مثلاً من مسودة لمصروف مباشرة)
- منع حذف مورد له أوامر شراء مسجلة (نفس نمط الحماية المتبع في باقي النظام)
- API: `/api/suppliers` (+`/[id]`) `/api/purchase-orders` (+`/[id]` للعرض وتحديث الحالة)
- شاشات: **الموردون** (قائمة + إضافة/تعديل/حذف)، **أوامر الشراء** (قائمة + إضافة ببنود متعددة + أزرار الـ workflow)
- دالة `computePoTotal` في `src/lib/purchase-orders.ts` مغطاة باختبارات Vitest

## ما تم بناؤه — Module 4: المخازن
- قاعدة بيانات: `Warehouse` (مخازن متعددة)، `InventoryItem` (أصناف بوحدة وحد إعادة طلب)، `StockMovement` (حركة وارد/منصرف، مرتبطة بمخزن وصنف ومشروع اختياريًا)
- **الرصيد بيُحسب من الحركات مباشرة** (مفيش حقل "رصيد" مخزّن يتعارض مع الحركات الفعلية) — دالة `computeStockBalance` في `src/lib/inventory.ts`
- **منع الرصيد السالب**: أي حركة "منصرف" بتتحقق أول إن الرصيد الحالي يكفي قبل ما تتسجل، وترجع رسالة خطأ واضحة لو مش كافي
- تنبيه بصري "مخزون منخفض" لما الرصيد يوصل لحد إعادة الطلب أو أقل
- API: `/api/warehouses` `/api/inventory-items` `/api/stock-movements`
- شاشة **المخازن**: جدول أصناف بالأرصدة الحالية وحالة كل صنف، + فورمات إضافة مخزن/صنف/حركة
- اختبارات Vitest كاملة لحساب الرصيد، التحقق من إمكانية الصرف، وكشف المخزون المنخفض

## ما تم بناؤه — Module 5: المعدات
- قاعدة بيانات: `Equipment` (حالة: متاحة/بمشروع/بالصيانة/خارج الخدمة)، `EquipmentAssignment` (تاريخ تخصيص المعدة للمشاريع، من/إلى تاريخ)، `MaintenanceLog` (سجل صيانة بالتكلفة والوصف)
- **منطق حالة محكوم بقواعد**: معدة تتخصص لمشروع بس لو "متاحة"، تدخل صيانة من أي حالة إلا لو خارج الخدمة، وعند إنهاء التخصيص بتختار لو محتاجة صيانة (تروح MAINTENANCE) أو لأ (ترجع AVAILABLE) — منطق `src/lib/equipment.ts` مغطى باختبارات كاملة
- API: `/api/equipment` (+`/[id]`, `/[id]/assign`, `/[id]/unassign`, `/[id]/maintenance`)
- شاشة **المعدات**: جدول بالحالة والمشروع الحالي، وأزرار سياقية حسب الحالة (تخصيص/إنهاء تخصيص/صيانة/إنهاء صيانة)

## ما تم بناؤه — Module 6 & 7: الموارد البشرية والرواتب + الحضور والانصراف
- قاعدة بيانات: `Employee` (بيانات الموظف، راتب أساسي)، `AttendanceRecord` (حضور/غياب/إجازة/مرضي لكل يوم، unique بالموظف والتاريخ)، `PayrollRecord` (راتب شهري لكل موظف، unique بالموظف والشهر)
- **توليد الراتب تلقائي بالكامل**: صافي الراتب = الأساسي + البدلات − (الخصومات الإضافية + خصم الغياب المحسوب تلقائيًا من سجلات الحضور الفعلية للشهر). منطق الحساب في `src/lib/payroll.ts` (صافي الراتب، خصم الغياب النسبي، ملخص الحضور، ساعات العمل) مغطى باختبارات كاملة
- Workflow الراتب: مسودة → معتمد → مصروف (صرف يتطلب Admin)، بمنع أي قفزة حالة
- شاشة **الحضور والانصراف**: تسجيل سريع بضغطة واحدة لكل موظف يوميًا (حاضر/غائب/إجازة/مرضي)
- شاشة **الرواتب**: توليد راتب لموظف عن شهر معين، جدول بكل رواتب الشهر مع الإجمالي، وأزرار الاعتماد والصرف
- شاشة **الموظفون**: قائمة + إضافة + تفعيل/تعطيل
- API: `/api/employees` (+`/[id]`) `/api/attendance` `/api/payroll` (+`/[id]`)

## ما تم بناؤه — تقارير تنفيذية
صفحة `/reports` دلوقتي فيها تبويب **تقارير تنفيذية** كنظرة شاملة عبر كل الموديولات السبعة مرة واحدة: مشاريع جارية، عقود، مستخلصات معلّقة اعتماد، أوامر شراء مفتوحة وقيمتها، أصناف مخزون منخفضة، حالة المعدات، موظفين نشطين، إجمالي رواتب الشهر، موردين نشطين، وعدد الشركاء — كل ده بيانات حقيقية من الـ APIs الفعلية مش أرقام وهمية.

## نقطة مهمة عن التوسع لباقي موديولات الـ ERP
المواصفات الأصلية اقترحت إضافة موديولات مستقبلية كتير. اللي اتبنى لحد دلوقتي: **الأرباح والشركاء**، **العقود والمستخلصات**، **الموردين والمشتريات**، **المخازن**، **المعدات**، **الموارد البشرية والرواتب**، **الحضور والانصراف**، وتقارير تنفيذية شاملة. الباقي: أوامر إسناد/تغيير، إدارة وثائق، إدارة مخاطر، إدارة جودة — حوالي **4 موديولات تانية** بس دلوقتي.

قاعدة البيانات والمعمارية (Modular Monolith) مصممة من الأول عشان تستوعبهم بسهولة — كل موديول جديد = Prisma models جديدة + مجلد `src/app/<module>/` + `src/app/api/<module>/` منفصل، من غير أي تعديل في اللي موجود. الأنسب نبنيهم واحد واحد حسب الأولوية.

## النشر على Vercel (بدون ما يأثر على موقعك القديم)
1. اعمل ريبو جديد على GitHub اسمه `section-one` وارفع الكود:
   ```bash
   git init
   git add .
   git commit -m "Section One - initial scaffold"
   git branch -M main
   git remote add origin https://github.com/<your-username>/section-one.git
   git push -u origin main
   ```
2. من Vercel Dashboard: **New Project** → استورد ريبو `section-one`.
3. في تبويب **Storage** بمشروع Vercel: أضف **Vercel Postgres** — هيضيف `POSTGRES_PRISMA_URL` و`POSTGRES_URL_NON_POOLING` تلقائيًا للـ Environment Variables.
4. ضيف Environment Variable إضافي: `NEXTAUTH_SECRET` (تقدر تولّده بـ `openssl rand -base64 32`) و`NEXTAUTH_URL` = دومين المشروع.
5. اعمل Deploy.
6. اربط subdomain جديد من نفس الدومين بتاعك من غير ما تلمس الموقع القديم:
   - من Vercel → Project Settings → Domains → أضف `app.section-eg.com`
   - من DNS إدارة الدومين (Google Domains / اللي انت شغال بيه): أضف CNAME record باسم `finance` يشاور على `cname.vercel-dns.com`
   - الموقع القديم وباقي السجلات (root domain، www، الـ ERP القديم) تفضل زي ما هي بدون أي تغيير.

## ملاحظة معمارية مهمة
القاعدة صممت عشان لما تحب تضيف موديول جديد (مثلاً Contracts) تعمل:
- Prisma models جديدة في نفس `schema.prisma` (أو تقسيمه لملفات متعددة لاحقًا بـ multi-file schema)
- مجلد جديد `src/app/contracts/` + `src/app/api/contracts/`
- من غير أي تعديل في الموديول الحالي
